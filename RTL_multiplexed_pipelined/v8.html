Simple Code-to-Block Map

A quick intuitive guide: "when you see this code section, it means this hardware function".

A. One-Line Mental Model

valid_in enters a smart window, then three compute blocks (L1, L2, L3), and finally emits valid_out with out_I/out_Q.

Input Window

Build 5-tap context, manage warm-up and flush.

L1 Engine

10 -> 32 with 4-tick MAC sweep.

L2 Engine

32 -> 32, A/B halves run in parallel over 6 ticks.

L3 Engine

32 -> 2, final saturated outputs over 6 ticks.

Output Write

Testbench stores only cycles where valid_out=1.

2. LAYER 1
signals: l1_tick, l1_busy, l1_latched_*		 Feature extractor Runs 4-cycle MAC schedule over 10 scalar inputs to generate 32 activations. l1_out[0:31], v_l1
3. LAYER 2
signals: l2_tick, l2_busy, L2A/L2B ROMs		 Hidden transform Processes first 16 and second 16 inputs in parallel each tick (not sequential halves). l2_out[0:31], v_l2
4. LAYER 3
signals: l3_tick, l3_busy, L3A/L3B ROMs		 Output head Collapses 32 features into 2 outputs; applies saturation and emits final valid pulse. out_I, out_Q, valid_out

F. Fast Debug Rules (Intuitive)
No output? Check if v_win ever pulses first.
Wrong timing? Trace pulse order: v_win -> v_l1 -> v_l2 -> valid_out.
Wrong amplitude? Verify weight order in packed 48-bit ROM words.
Tail mismatch? Confirm flush timeout and drain cycles are long enough.
Random garbage at start? Re-check memory init and reset sequence.
