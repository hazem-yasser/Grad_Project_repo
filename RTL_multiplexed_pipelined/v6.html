DECODING THE MONOLITHIC PIPELINE

An exhaustive breakdown of the Time-Multiplexed SystemVerilog FSM, Hex Formatting, and Deep Search Python Verification.

1. THE SYSTEMVERILOG MULTIPLEXER (THE "CASE" STATEMENT)

What is this code doing?

In the monolithic design, Layer 3 Stage B needs to process 32 inputs. However, we only have 3 multipliers (a Triple-MAC) to save silicon area. To process 32 inputs with only 3 multipliers, we must reuse them over time. This is called Time-Multiplexing.

The tick_cnt acts as the conductor of an orchestra. As the clock ticks from 1 to 6, the case statement routes different chunks of the pipeline register (l2b_reg) into the multiplier inputs (m_in1, m_in2, m_in3).

always_comb begin case (tick_cnt) 1: {m_in1, m_in2, m_in3} = {l2b_reg[16], l2b_reg[17], l2b_reg[18]}; 2: {m_in1, m_in2, m_in3} = {l2b_reg[19], l2b_reg[20], l2b_reg[21]}; // ... cycles 3, 4, 5 do the same for the next inputs ... 6: {m_in1, m_in2, m_in3} = {l2b_reg[31], 16'sd0, 16'sd0}; default: {m_in1, m_in2, m_in3} = {16'sd0, 16'sd0, 16'sd0}; endcase m_w = (mac_en) ? L3B_W_ROM[(i * 6) + rom_idx] : 48'd0; end

Tick 1 to 5 (Heavy Lifting)

The Triple-MAC consumes 3 inputs per clock cycle. Over 5 cycles, it processes 15 inputs (Indices 16 through 30).

Tick 6 (The Zero Pad)

We only have 1 input left (Index 31). The MAC unit demands 3. We feed it Index 31, and force the other two inputs to 16'sd0. Multiplying by zero safely ignores the unused multipliers.

The ROM Index Math

(i * 6) + rom_idx ensures each of the 2 neurons (i) gets its specific chunk of 6 weights, perfectly synchronized with the `tick_cnt`.

2. PIPELINE PACING & VALID OUT

// Output assignment at the bottom of the module assign valid_out = (tick_cnt == 8 && fill_cnt == 6);

The Conductor's Baton

Why exactly cycle 8? And why fill count 6?

tick_cnt == 8: The MAC units finish multiplying at cycle 6. Cycle 7 is used to add Stage A and Stage B together. Cycle 8 is the "Commit" cycle, where the answer is finally written to the output register out_I and out_Q. Therefore, the data is only stable and valid on exactly cycle 8 of every 9-cycle period.

fill_cnt == 6: When you hit the reset button, the pipeline is full of garbage (or zeros). It takes 3 shifts for the first real input to reach the center of the window. Then it takes 3 more shifts for that data to propagate through Layer 1, Layer 2, and Layer 3. Total latency = 6 symbol periods. The valid_out pin is artificially forced to 0 until the 6th period to prevent spitting out garbage data.

3. PYTHON: HEX PREPARATION & TARGET EXTRACTION

Converting Arrays to Hardware-Ready Streams

Hardware cannot read raw Python 2D arrays. It needs continuous streams of binary/hexadecimal numbers. This script does two critical things: Packing weights for the Triple-MAC, and flattening the input stimulus.

save_triple_width_hex()

This function takes the floating-point weights, scales them to integer Q1.14, converts them to 4-character hex (16-bit), and then concatenates 3 of them together into a 48-bit string (e.g., 0A2B 11FF C012). This is because your Triple-MAC expects 3 weights simultaneously on a single wire.

Target Extraction (Index 2 & 7)

The Golden Model generates a 10-value block representing a sliding window. However, the true "target" of the neural network is just the center symbol. Index 2 is the center real part (I), and Index 7 is the center imaginary part (Q). The script discards the redundant window history, sending only a pure stream of centers to the RTL.

4. PYTHON: THE DEEP SEARCH VERIFIER

The Alignment Problem

If you have an array of ideal outputs [A, B, C, D] and an array of RTL outputs [0, 0, 0, A, B, C, D], you cannot simply subtract them to find the error. The RTL has a hardware delay. If you compare Target index 0 to RTL index 0, the error will be massive.

The Sweep Loop

The script uses a for delay in range(-80, 91): loop. It acts like a combination lock, shifting the RTL array across the Target array one index at a time. At every step, it calculates the Mean Squared Error (MSE). The delay that produces the lowest MSE is mathematically proven to be the correct pipeline latency.

# The Sub-Array Slicing Magic if delay > 0: t_slice = target_complex[delay : delay + len(r_complex)] r_slice = r_complex[:len(t_slice)]

This code prevents array out-of-bounds errors. If the best delay is 6, it chops off the first 6 targets (target_complex[6:]), perfectly aligning Target 6 with RTL 0. It then cuts both arrays to the exact same length (len(t_slice)) so NumPy can subtract them element-by-element.
